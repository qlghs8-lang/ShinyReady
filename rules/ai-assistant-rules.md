# 🤖 AI Assistant Rules — Shiny Ready!

## 🎯 핵심 원칙

**명령한 내용과 함수만 수정하도록 한다.**
**명령한 것 이외의 것들은 무조건 질문한다.**
**함부로 지우거나 초기화하지 않는다.**
**기존 구조를 해치지 않는다.**
**수정 이후에 큰 변동이나 문제가 발생할 수 있는 경우는 문제를 미리 예측하여 보고하고 수정 전에 반드시 질문을 한다.**
**기존 코드들의 규칙과 구조를 레퍼런스로 하여 일관성을 유지한다.**

---

## 🎯 **작업 범위 제한**

### ✅ **수정 가능한 범위**

- 명시적으로 요청받은 특정 함수
- 명시적으로 요청받은 특정 파일
- 명시적으로 요청받은 특정 코드 블록
- **기존 코드 패턴과 일치하는 구조적 개선** (기존 코드를 레퍼런스로 활용)

### ❌ **수정 금지 범위**

- 요청하지 않은 다른 함수
- 요청하지 않은 다른 파일
- 자동 최적화하는 리팩토링
- 추가적인 기능 구현
- **기존 씬(Scene) 파일 구조 변경** (클라이언트 호환성 문제)
- **기존 Prefab 구조 변경** (참조 깨짐 위험)
- **기존 API 인터페이스 변경** (호출부 호환성 문제)

---

## 📋 **안전한 작업 방식**

### 1. 명령 분석

- 요청받은 정확한 내용 파악
- 수정할 파일과 함수 식별
- 범위 명확히 설정

### 2. 작업 실행

- 지정된 부분만 변경
- 다른 코드는 건드리지 않음
- 원본 구조 유지
- **기존 코드 스타일과 패턴 참조하여 일관성 유지**

### 3. 결과 보고

- 수정된 내용만 보고
- 변경 사항 명확히 설명
- 추가 제안은 별도로 안내

---

## ❓ **질문 필요 규칙**

### 🔔 **질문이 필요한 상황**

- 명령 범위를 벗어나는 작업이 필요한 경우
- 추가적인 최적화나 개선이 가능한 경우
- 관련된 다른 파일이나 함수에 영향이 있는 경우
- 보안이나 성능 이슈가 발견된 경우
- 코드 품질 개선이 필요한 경우

### 💬 **질문 방식**

- **명확한 질문**: 구체적인 내용과 이유 제시
- **선택지 제공**: 가능한 옵션들을 명시
- **영향도 설명**: 변경 시 예상되는 영향 설명
- **승인 대기**: 사용자 확인 후 진행

### ⏸️ **작업 중단**

- 명령 범위를 벗어나는 작업 발견 시 즉시 중단
- 사용자에게 상황 설명 및 질문
- 승인 하에만 추가 작업 진행

---

## ⚠️ **위험 예측 및 사전 보고 규칙**

### 🔮 **위험 예측 필요 상황**

- **씬(Scene) 파일 수정**: GameObject 참조 깨짐 위험
- **Prefab 구조 변경**: 인스턴스 참조 깨짐 위험
- **ScriptableObject 스키마 변경**: 기존 에셋 데이터 손실 위험
- **핵심 비즈니스 로직 변경**: 게임플레이 동작 변화
- **의존성 라이브러리 변경**: 호환성 문제
- **파일 구조 변경**: 경로 참조 깨짐
- **MonoBehaviour 컴포넌트 제거/이름변경**: 씬에서 Missing Script 발생

### 📊 **위험도 분류**

- **높음 (High)**: 전체 시스템에 큰 영향
- **중간 (Medium)**: 특정 모듈에 영향
- **낮음 (Low)**: 제한적인 영향

### 🚨 **사전 보고 절차**

1. **위험 요소 식별**: 작업 전 발생 가능한 문제점 분석
2. **영향도 평가**: 다른 파일, 기능, 시스템에 미치는 영향
3. **대안 제시**: 안전한 수정 방법이나 대안 옵션
4. **사용자 확인**: 위험도와 대안을 설명하고 승인 요청
5. **단계적 실행**: 승인 후 단계적으로 작업 진행

### 💡 **위험 예측 보고서 형식**

```
🚨 위험 예측 보고서

📝 수정 내용: [수정할 내용]
⚠️ 예상 위험: [발생 가능한 문제점]
📊 위험도: [High/Medium/Low]
📁 영향받는 파일: [파일 목록]
🎮 영향받는 기능: [기능 목록]
💡 대안 방식: [안전한 수정 방법]
✅ 확인 사항: [사용자 확인 요청]
```

- **영향도 분석**: 삭제 시 영향받는 다른 파일이나 기능 분석
- **최종 확인**: 삭제 실행 전 최종 확인 요청

## 🏎️ **Unity 모바일 최적화 원칙**

### 성능 우선 규칙
1. `Update()`에서 `GetComponent<>()` 호출 금지 → Awake/Start에서 캐싱
2. 빈번히 생성·삭제되는 오브젝트는 Object Pooling 사용
3. `FindObjectOfType()` 남용 금지 → 직접 참조 또는 싱글톤 사용
4. 문자열 연산 최소화 (StringBuilder 또는 상수 사용)
5. 물리 연산은 `FixedUpdate()`에서 처리
6. 코루틴 남용 주의 → 단순 딜레이는 `Invoke` 활용 고려

### 코딩 컨벤션
- **클래스**: PascalCase (예: `PlayerController`)
- **메서드**: PascalCase (예: `MovePlayer()`)
- **필드(private)**: camelCase + 언더스코어 (예: `_moveSpeed`)
- **프로퍼티**: PascalCase (예: `MoveSpeed`)
- **상수**: ALL_CAPS (예: `MAX_SPEED`)
- **SerializeField**: private 필드에 `[SerializeField]` 어트리뷰트 사용

3. **사전 보고**: 위험도와 대안을 사용자에게 보고
4. **사용자 확인**: 승인 후 진행
5. **범위 설정**: 수정할 정확한 위치 식별
6. **안전성 검사**: 삭제나 초기화가 필요한지 확인
7. **작업 실행**: 지정된 부분만 변경
8. **검증**: 요청 범위 내에서만 수정되었는지 확인
9. **질문**: 추가 작업이 필요한 경우 사용자에게 질문
10. **보고**: 수정 결과만 보고

## 🚫 **불필요한 추가 금지 규칙**

### ❌ **추가 금지 사항**

- **명시적으로 요청하지 않은 기능 구현**
- **과도한 최적화하는 리팩토링**
- **스키마에 정의되지 않은 필드 사용**
- **불필요한 복잡성 추가**
- **요청 범위를 벗어나는 개선 사항**
- **추가적인 유효성 검사나 로직**
- **명시적으로 요청하지 않은 에러 처리 강화**
- **과도한 로깅이나 디버깅 코드 추가**

### ✅ **최소화 작업 원칙**

- **요청받은 기능만 구현**
- **기존 코드 구조 유지**
- **필수적인 변경사항만 적용**
- **단순하고 명확한 로직 유지**
- **기존 코드베이스의 패턴과 컨벤션을 레퍼런스로 활용**

## ⚠️ **주의사항**

- **자동 최적화 금지**: 성능 개선이나 코드 정리는 요청 시만
- **확장 금지**: 요청 범위를 넘어서는 작업 금지
- **추가 기능 금지**: 명시적으로 요청하지 않은 기능 구현 금지
- **불필요한 추가 금지**: 스키마에 없는 필드나 과도한 기능 추가 금지
- **최소화의 작업 원칙**: 요청받은 기능만 구현하고 불필요한 복잡성 추가 금지
- **질문 필수**: 명령 범위 외 작업은 반드시 질문
- **삭제 금지**: 명시적 요청 없이 삭제나 초기화 금지
- **위험 예측 필수**: 큰 변동이나 문제 발생 가능성 사전 분석
- **구조 보호**: 기존 파일 구조와 API 인터페이스 변경 금지
